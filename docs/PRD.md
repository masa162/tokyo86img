# プロジェクト要件定義書 (PRD)

## 1. 開発パラダイムと責任範囲
本プロジェクトは、機能ごとにリポジトリを明確に分離し、管理の集中化とフロントエンドの純粋化を徹底する。

### リポジトリ構成と役割
1.  **unbelong.git (セントラル・リポジトリ)**
    - **役割**: バックエンドAPIの提供および**全フロントエンドの管理機能の集約**。
    - **構成要素**:
        - `apps/api`: 全サービス共通のバックエンドAPI。D1データベースを操作する唯一の窓口。
        - `apps/admin`: **唯一の管理画面 (Dashbord)**。イラスト、コミック、作品、画像アセットの全ての管理・投稿・編集はここで行う。
        - `apps/img-cdn`: 画像配信高速化のためのWorker。

2.  **unbelongillust.git (イラスト公開リポジトリ)**
    - **役割**: イラストの閲覧に特化したフロントエンド。
    - **制約**: **管理機能 (Admin/Dashboard) を一切含めてはならない**。読み取り専用のAPIを叩き、表示するのみ。

3.  **unbelongcomic.git (コミック公開リポジトリ)**
    - **役割**: コミック（Webtoon）の閲覧に特化したフロントエンド。
    - **制約**: **管理機能 (Admin/Dashboard) を一切含めてはならない**。ビューアーの実装に集中する。

## 2. 実装方針
- 各リポジトリの独立性を保つこと。
- フロントエンド（illust/comic）において、投稿や編集が必要になった場合は、必ず `unbelong.git` 内の管理画面にリンクするか、そちらで実装を行う。
- フロントエンドプロジェクトを生成する際は、管理画面のテンプレートやコンポーネントを混入させない。

## 3. 運用ルール
- 開発者は、機能の追加場所（管理系か閲覧系か）を厳格に判断してから実装を開始すること。
- リポジトリを跨ぐ共通の変数名や環境変数は、混乱を避けるため一貫した命名規則に従う。
- **セキュリティの徹底**: 認証情報やAPIのURLをソースコードに直接記述（ハードコード）してはならない。必ず環境変数を使い、本番環境での変数の流し込みは Cloudflare の管理パネルから手動で行うこと。

## 4. コア・コンセプト：植物観察モデル
「間違いを恐れずに、気軽にアップして、いらなくなったら消す」という運用思想。
- **自動化**: アップロード時にリソース最適化、フォーマット変換（WebP）、ショートURL生成を自動化。
- **安心感**: 管理画面からD1とCloudflare Imagesの両方を完全に同期して削除できる「一撃削除」の実装。
- **効率性**: 大量画像（Webtoon）のバッチ管理と、一括Markdown生成による「考えない投稿」の実現。
