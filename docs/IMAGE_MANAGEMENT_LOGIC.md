# 画像運用ロジック定義書（単品 vs 箱/複数）

## 1. 概要
本プロジェクトでは、用途に応じて「画像そのもの（単品）」と「画像を格納するコンテナ（箱/複数）」の2種類の運用コンテキストを使い分ける。

## 2. 運用コンテキストの分類

### A. 単品画像（一対一運用）
- **主な用途**: イラストギャラリー (`unbelongillust`)
- **データ構造**: `Illustration` エンティティ
  - `image_id`: Cloudflare Images の ID（これが主役）
  - `title`, `description`: 画像に付随するメタデータ
- **特徴**: 1枚の画像が1つの「作品」として独立している状態。

### B. 箱・複数画像（一対多運用 / コンテナ運用）
- **主な用途**: コミック・マンガ (`unbelongcomic`)
- **データ構造**: `Episode` エンティティ
  - `content`: **画像 ID の配列を JSON 文字列化したもの**（例: `["id1", "id2", "id3"]`）
  - `thumbnail_image_id`: 一覧表示用の表紙画像
- **特徴**: `Episode` という「箱（コンテナ）」の中に、複数の画像が特定の順序（ページ順）で格納されている状態。ビューアーはこの配列を読み込み、縦に並べて表示する (Webtoon 形式)。

## 3. データの持ち方と拡張性
- `Illustration` エンティティにも `content` フィールドが存在するため、将来的に「複数枚で構成されるイラストセット」をこのコンテナ運用へ拡張することが可能。
- `Episode` の `content` は現状 **JSON 文字列** として処理する。将来的にテキストやマークアップを混在させる場合も、パースロジックを共通化して対応する。

## 4. 管理上のルール
- **アップロード**: Cloudflare Images へアップロードし、得られた ID を `unbelong-api` 経由で D1 に保存する。
- **単品登録**: `unbelong-api` のイラスト投稿エンドポイントを使用し、`image_id` に 1つの ID をセットする。
- **箱（複数）登録**: `unbelong-api` のエピソード投稿エンドポイントを使用し、`content` に ID 配列をシリアライズしてセットする。
